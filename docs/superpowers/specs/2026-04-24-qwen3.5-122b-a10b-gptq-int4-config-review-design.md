# qwen3.5-122b-a10b-gptq-int4 config review design

## Goal

Create a small source-artifact pair under `docs/reviews/` for the local model directory `/mnt/md127/LLM/Qwen3.5-122B-A10B-GPTQ-Int4`:

1. a mirrored copy of the model's `config.json`
2. a detailed markdown review that explains both the model structure and what the config implies for this repo's `serve` path

The result should help a reader answer two questions from one place:

- what this `config.json` says about the model itself
- how those settings matter when wiring the model into the current repo's vLLM-backed serve flow

## Chosen output

- **Mirrored source file:** `docs/reviews/qwen3.5-122b-a10b-gptq-int4-config.json`
- **Review document:** `docs/reviews/qwen3.5-122b-a10b-gptq-int4-config-review.md`
- **Primary audience:** both a learner trying to understand the model and a maintainer trying to reason about serve integration

## Scope

Include only information grounded in the checked source files and already-known repo behavior.

### In scope

1. copy the source `config.json` without modification
2. explain top-level fields in the copied JSON
3. explain the language-model architecture signals in `text_config`
4. explain the multimodal signals in `vision_config` and token IDs
5. explain the quantization settings in `quantization_config`
6. connect those fields to the current repo's serve wrapper behavior:
   - `ServeConfig.model_id`
   - `extra_args`
   - unconditional `--enable-lora`
   - `tensor_parallel_size`
   - startup timeout and memory pressure considerations
7. provide a minimal YAML example for this repo and a more realistic multi-GPU example
8. optionally add the new review doc to `docs/reviews/README.md` and `docs/reviews/manifest.json` if that keeps the review-doc surface consistent

### Out of scope

- editing core Python serve code
- modifying the source model directory
- claiming the model fully runs on this host unless separately verified by an actual serve test
- exhaustive benchmark reproduction from the vendor README
- training-path guidance

## Structure

The markdown review should use this structure:

1. `# qwen3.5-122b-a10b-gptq-int4 config review`
2. `## What this file is`
3. `## Source artifact`
4. `## Top-level reading guide`
5. `## Language-model architecture signals`
6. `## Vision and multimodal signals`
7. `## Quantization signals`
8. `## What this means for this repo's serve path`
9. `## Minimal YAML examples`
10. `## Practical caveats`
11. `## Short summary`

## Content rules

- The copied JSON must remain byte-for-byte semantically unchanged.
- The review must distinguish clearly between facts from `config.json`, facts from the model README, and repo-specific interpretation.
- The report should explain unfamiliar fields in plain language first, then tie them back to serving implications.
- The report should stay grounded in files already read:
  - `/mnt/md127/LLM/Qwen3.5-122B-A10B-GPTQ-Int4/config.json`
  - `/mnt/md127/LLM/Qwen3.5-122B-A10B-GPTQ-Int4/README.md`
  - `src/aiinfra_e2e/config.py`
  - `src/aiinfra_e2e/serve/vllm_server.py`
- If the review mentions suggested CLI/YAML flags, it must make clear whether they come from the repo wrapper or from upstream model/runtime guidance.
